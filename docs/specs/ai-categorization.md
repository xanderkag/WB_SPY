# ТЗ: AI-категоризация названий товаров

> Статус: **draft** · приоритет: P2 · оценка: **4–6 часов**

## Зачем

Сейчас название товара — длинная строка вида:

> *«Ноутбук MacBook Air 13 (2025), 16 512Gb, (MC6U4), Серый»*

Человек оттуда видит: год, диагональ, RAM, SSD, артикул производителя, цвет.
Машина — только бренд (через `subject_id`). Из-за этого:

- **Каталог нельзя нормально фильтровать.** Хочешь «MacBook Air 13″ 16GB до 100К» — никак.
- **Дедупликация невозможна.** Один и тот же MacBook M5 16/512 в трёх цветах = три карточки в нашей БД, не понятно что это один продукт.
- **Сравнение цен ломается.** Без артикула (`MC6U4`) трудно понять что у двух продавцов один и тот же товар.

AI-категоризация **разбирает строку на атрибуты**: `{year, diagonal, ram, ssd, chip, color, sku, type}`.

## Что переиспользуем

- `products.name` — источник.
- Уже работающая FastAPI + SQLite — храним результат.
- Любой LLM API (OpenAI / Anthropic / локальный Ollama) — для извлечения.

## Подход: гибрид regex + LLM

**Regex-экстрактор** ловит простые паттерны:
- диагональ — `\b(\d{2})["”]?(?=\s*\(|$| )` → 13, 14, 16
- RAM — `\b(\d{1,3})\s*Gb\b` или `\b(\d{1,3})/(\d{3,4})Gb\b`
- SSD — то же, второе число
- год — `\((\d{4})\)`
- SKU производителя — `\(([A-Z]{2,3}\d{2,4}[A-Z]?\d?)\)` (MC6U4, MGEE4 и т.п.)

**LLM** добивает остальное (чип, цвет, тип, точное имя модели) — там где regex слабоват:
- одним запросом батчит 50 названий
- модель: `gpt-4o-mini` (~$0.15 / 1M input) или `claude-haiku` (~$0.25 / 1M input)
- 131 товар × ~80 токенов = ~10К токенов = **~$0.001 за полный backfill**

Кэшируем: один раз на `nm_id`. Перекатегоризация — только по форс-команде.

## Схема БД

Новая таблица:

```sql
CREATE TABLE IF NOT EXISTS product_attrs (
  nm_id     INTEGER PRIMARY KEY,
  year      INTEGER,
  diagonal  REAL,         -- 13.0, 14.0, 16.0
  ram_gb    INTEGER,      -- 16, 24, 48
  ssd_gb    INTEGER,      -- 256, 512, 1024, 2048
  chip      TEXT,         -- "M5", "M5 Pro", "M5 Max"
  color     TEXT,         -- "Black", "Silver"
  sku       TEXT,         -- "MGEE4" (артикул производителя)
  model     TEXT,         -- "MacBook Pro 16" (без года/конфига)
  type      TEXT,         -- "laptop", "smartphone", "headphones"
  source    TEXT,         -- "regex" | "llm" | "manual"
  updated_at INTEGER NOT NULL
);
CREATE INDEX idx_attrs_model ON product_attrs(model);
CREATE INDEX idx_attrs_chip  ON product_attrs(chip);
```

## API

- `GET /api/products` → каждый товар получает поле `attrs: {…}`.
- `GET /api/filters/options` → `{models: ["MacBook Pro 16", ...], chips: [...], ram: [16,24,48], ...}` для UI-чипов.
- `POST /api/categorize/{nm_id}` — форс-перекатегоризация одного товара.
- `POST /api/categorize/backfill` — прогнать все товары без `attrs` (admin-only).

## Worker hook

В `collector.py` после `upsert_products()`:
- собираем nm_id, у которых ещё нет записи в `product_attrs`
- batch (по 50) → regex_pass → если `chip|color|model` не извлеклись, добавляем в LLM-batch
- LLM-вызов раз в тик (раз в 30 мин), даже если новых товаров пара штук — экономнее.

## UI

В **каталоге** над списком — горизонтальный скролл фильтров-чипов:

```
[модель: MacBook Pro 16 ↓] [RAM: 16+ ↓] [SSD: 512+ ↓] [цвет ↓] [✕ сброс]
```

В **детальной модалке** — блок «Характеристики»:

```
🪪 MGEE4 · M5 Max · 16″ · 48/2TB · Black · 2025
```

В **админке** (точечные товары): таблица показывает не «Ноутбук MacBook 16 Pro (M5 Max), 48 2Tb, (MGEE4), Black», а понятный ярлык `MacBook 16 Pro M5 Max · 48/2TB · Black`.

## Открытые вопросы

1. **Провайдер LLM.** OpenAI или Anthropic или локальный Ollama (`llama3.1:8b-instruct`)?
   - Anthropic: уже есть подписка пользователя
   - OpenAI: дешевле для batch
   - Локальный: бесплатно, но требует Ollama-инстанса
2. **Схема атрибутов** — закрытая (как выше, 9 полей) или открытая (LLM сам решает что важное)?
   - **Рекомендую закрытую** — фильтры в UI должны иметь известный список значений.
3. **Backfill** — сразу при запуске? Или по команде?
   - **Рекомендую по команде** (`POST /api/categorize/backfill`) — чтобы случайно не сжечь токены.

## Риски

- LLM может галлюцинировать атрибуты которых в названии нет → валидировать в коде против regex-вытащенных.
- Для незнакомых брендов (китайские наушники без структуры) категоризация будет шумной → `attrs` могут быть `null`, фильтры это нормально пропускают.

## Этапы

2. **Phase 2 (2ч):** LLM-fallback для несхваченных полей + worker hook.
3. **Phase 3 (1ч):** backfill-команда + админка-кнопка «прогнать всё заново».
