# ТЗ: Профиль продавца + heatmap «когда падают цены»

> Статус: **draft** · приоритет: P2 · оценка: **4–5 часов**

## Зачем

Сейчас в админке тапнул на R2D2 — увидел список товаров. Полезно, но это
просто каталог. **Реальная инсайтовая ценность** — паттерны:

> *R2D2 обычно дропает цены по средам и пятницам с 18:00 до 22:00.*
> *Средняя глубина скидки — 5.2%. Средняя длительность — 14 часов.*
> *Самые волатильные товары — Apple MacBook M5 Max.*
> *За последние 30 дней самая глубокая скидка была −11% (10.07.2026).*

Это даёт пользователю **поведенческое понимание продавца** — когда стоит
заходить чекать, чего ждать. Сейчас уже глазами видно что R2D2 двигает цены
пакетно вечером, но в один взгляд это не очевидно.

## Что переиспользуем

- `price_snapshots` — полная история, одна строка на каждое наблюдение.
- `products` + `tracked_suppliers` — связь товаров с продавцами.
- `alerts` — уже сработавшие дропы (бонусный источник).

## Метрики профиля

| метрика | как считается |
|---|---|
| **Heatmap (час × день недели)** | сетка 7×24, в каждой ячейке — сколько было ценовых движений (любых: вверх/вниз) |
| **Heatmap (только дропы)** | то же, но только когда `price_t < price_{t-1}` |
| **Средняя глубина скидки** | для каждого «эпизода скидки» считаем `(до − дно) / до * 100`, усредняем |
| **Частота переоценок в сутки** | `total_changes / days_tracked` |
| **Топ часы активности** | top-3 часовых слота с наибольшей `count(changes)` |
| **Средняя длительность скидки** | время от первого падения до возврата к до-скидочному уровню |
| **Топ-волатильные товары** | по `stddev(sale_price) / mean(sale_price)` за период |
| **Самая глубокая скидка** | `max(drop_pct)` по всем `alerts` этого продавца |

## SQL-расчёты (черновики)

**Heatmap движений** — одним проходом по снапшотам:

```sql
WITH deltas AS (
  SELECT
    nm_id, ts, sale_price,
    LAG(sale_price) OVER (PARTITION BY nm_id ORDER BY ts) AS prev_price
  FROM price_snapshots
  WHERE nm_id IN (SELECT nm_id FROM products WHERE supplier_id = ?)
)
SELECT
  CAST(strftime('%w', ts, 'unixepoch', 'localtime') AS INTEGER) AS dow,  -- 0=вс, 1=пн
  CAST(strftime('%H', ts, 'unixepoch', 'localtime') AS INTEGER) AS hour,
  COUNT(*)                AS changes,
  SUM(CASE WHEN sale_price < prev_price THEN 1 ELSE 0 END) AS drops,
  AVG(CASE WHEN sale_price < prev_price
           THEN (prev_price - sale_price) / prev_price * 100
           END)           AS avg_drop_pct
FROM deltas
WHERE prev_price IS NOT NULL AND sale_price != prev_price
GROUP BY dow, hour
ORDER BY dow, hour;
```

Возвращает до 168 ячеек (7×24), большинство нулевые.

**Топ-волатильные** — coefficient of variation:

```sql
SELECT
  p.nm_id, p.brand, p.name,
  COUNT(s.id) AS n,
  AVG(s.sale_price) AS mean_price,
  -- SQLite не имеет stddev; считаем через двойной проход в коде или
  -- SUM((price - mean)^2) / N; для PoC берём (max - min) / mean
  (MAX(s.sale_price) - MIN(s.sale_price)) / NULLIF(AVG(s.sale_price), 0) AS cv
FROM products p
JOIN price_snapshots s USING(nm_id)
WHERE p.supplier_id = ?
GROUP BY p.nm_id
HAVING n >= 10
ORDER BY cv DESC
LIMIT 5;
```

### Heatmap-компонент

SVG, 7 строк × 24 колонки.

- Размер ячейки: ~12×16px на мобиле, плотно.
- Цвет: `bg-slate-700` (пусто) → `bg-sky-700` (мало) → `bg-sky-400` (много) →
  `bg-amber-300` (максимум). Шкала логарифмическая.
- Tap по ячейке → tooltip «Среда 18:00 — 12 движений, ср. дроп −4.2%».
- Подписи: дни слева, часы внизу с шагом 3.

## Проблема: мало данных в начале

Сейчас у нас ~1.5 дня истории. **Honest UX**: если `days_tracked < 7`, на
странице профиля рисуем placeholder:

> *⏳ Копим историю.*
> *Профиль продавца станет видим через ~5 дней (нужно ≥7 дней наблюдений).*
> *Пока есть только базовые цифры.*

И показываем только то что считается на коротких данных (changes_per_day,
max_drop).

## Открытые вопросы

1. **Учитывать ли все товары или только активные «в наличии»?** Если товар
   `in_stock=0` — снимки могут быть «битые». Решение: фильтр `in_stock=1`
   при подсчёте deltas. (Но у нас `in_stock` сейчас всегда 0 для R2D2 —
   WB не отдаёт стоки на этом endpoint'е → можно не фильтровать.)
2. **Heatmap по UTC или локальному времени?** Для пользователя — локальное
   (MSK). Использовать `strftime('%H', ts, 'unixepoch', 'localtime')`.
3. **Окно по умолчанию.** 30 дней или 7? Меньше = свежее данные, но шумнее.
   **Рекомендую `?days=30` дефолт, но в UI можно переключать 7/30/90.**

## Риски

- Долгие SQL-запросы на больших объёмах. Сейчас 3к снапшотов — ничего
  страшного, но через год это будет ~2М строк на одного продавца. **Митигация:**
  индекс уже есть (`idx_snap_nm_ts`), при необходимости — материализованная
  view с агрегацией по часам.
- Шумный heatmap без больших объёмов истории. **Митигация:** placeholder при
  `days_tracked < 7`.
