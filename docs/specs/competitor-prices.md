# ТЗ: Сравнение цен у других продавцов WB

> Статус: **draft** · приоритет: P1 · оценка: **6–8 часов**

## Зачем

Сегодня мы знаем цены только наших `tracked_suppliers` (R2D2). Один и тот же
MacBook Pro 16 (MX313) на WB продают R2D2, ТехноПарк, Apple Shop и ещё десяток
магазинов — у каждого своя цена и своя динамика. Пользователю важно видеть:

> *MacBook Pro 16 (MX313)*
> *• R2D2 — 382 727 ₽ (наш) ← вы здесь*
> *• ТехноПарк — 392 100 ₽*
> *• Apple Shop — 401 500 ₽*
> *• ⭐ лучшая цена сейчас у нашего продавца*

Это превращает приложение из «следим за одним продавцом» в **«находим лучшую
цену на WB»** — кардинальный сдвиг ценности.

## Что переиспользуем

- `wbp/wb_cffi.py` — клиент к WB с TLS-Chrome импесонацией.
- `products` — наши товары (база с которой ищем «соседей»).
- `product_attrs.sku` (если уже есть из AI-категоризации) — артикул производителя,
  лучший якорь для дедупликации (см. ниже).

## Как идентифицировать «тот же товар»

Это главная сложность. У разных продавцов одинаковый MacBook может называться:

- *Ноутбук MacBook 16 Pro (M5 Max), 48 2Tb, (MGEE4), Black* (R2D2)
- *Apple MacBook Pro 16 M5 Max 48GB 2TB Black* (другой продавец)
- *MacBook Pro 16″ M5 Max 48/2TB Чёрный, MGEE4* (третий)

Якоря по убыванию надёжности:

| якорь | где взять | надёжность |
|---|---|---|
| `imt_id` | поле в JSON-ответе WB (внутренний market id) | **100%** — гарантия одного товара |
| `sku` производителя | из AI-категоризации (`MGEE4`) | 95% — может отсутствовать у части продавцов |
| brand + model + RAM/SSD/color | AI-категоризация | 80% — близкие конфиги путаются |
| близкое имя + цена в полосе ±20% | строковая близость | 50% — много ложных |

**План:** в первую очередь идти по `imt_id` (он точно есть в ответе WB) +
sanity-check `sku`. Если у нас нет AI-категоризации — использовать только `imt_id`.

## Источник данных

Нужен поисковый эндпоинт WB:

| endpoint | даёт | минус |
|---|---|---|
| `search.wb.ru/exactmatch/.../search?query=...` | список товаров по строке от любых продавцов | **под PoW**, на нашем VPN-IP режется. На проде может работать через нормальный VPN. |
| `catalog.wb.ru/catalog/<shard>/v4/catalog?xsubject=<id>` | категорийный листинг (без поиска) | soft-block на некоторых shard'ах |
| `card.wb.ru/cards/v2/detail?nm=...` | детали одного товара | TCP-drop на нашем IP |

**Fallback-стратегия:**
1. Пробуем `search.wb.ru` (на проде через нормальный VPN — ожидаем работает).
2. Если не идёт — обходим `catalog.wb.ru` ПО `imt_id` (есть endpoint
   `search/v2/preset?preset=<imt>` — даёт всех продавцов одного товара).
3. Hard-fallback: руками добавлять через админку «эти товары следить как
   конкурентов».

## Схема БД

```sql
CREATE TABLE IF NOT EXISTS competitor_offers (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  base_nm_id      INTEGER NOT NULL,    -- наш товар (на который смотрим)
  competitor_nm_id INTEGER NOT NULL,   -- конкурентский nm_id
  supplier_id     INTEGER,
  supplier_name   TEXT,
  brand           TEXT,
  name            TEXT,
  sale_price      REAL,
  ts              INTEGER NOT NULL,    -- момент снятия
  match_anchor    TEXT,                -- "imt_id" | "sku" | "manual"
  UNIQUE(base_nm_id, competitor_nm_id, ts)
);
CREATE INDEX idx_comp_base_ts ON competitor_offers(base_nm_id, ts);
```

История цен конкурентов хранится так же как наша основная — снапшотами,
не текущим состоянием. Чтобы можно было рисовать графики «как наш vs средний».

## API

- `GET /api/products/{nm_id}/competitors` →
  ```json
  {
    "ours":   {"supplier_name": "R2D2", "price": 382727, "is_best": true},
    "others": [
      {"supplier_name": "ТехноПарк",  "supplier_id": ..., "nm_id": ..., "price": 392100, "ts": ...},
      {"supplier_name": "Apple Shop", "supplier_id": ..., "nm_id": ..., "price": 401500, "ts": ...}
    ],
    "best_price": 382727,
    "median": 392100,
    "last_refresh_ts": 1781000000
  }
  ```
- `POST /api/products/{nm_id}/competitors/refresh` — форс-обновление одного товара.
- `GET /api/competitors/stats` — за сколько товаров мы нашли конкурентов,
  средний разрыв.

## Worker hook

В коллекторе **раз в N часов** (не каждый тик — нагрузка на WAF):

```python
async def refresh_competitors():
    if int(time.time()) % (6 * 3600) != 0:  # раз в 6 часов
        return
    for nm in await db.products_for_competitor_scan():
        offers = await wb_search_by_imt_or_name(nm)
        await db.upsert_competitor_offers(nm, offers)
        await asyncio.sleep(random.uniform(2, 5))  # бережно с WAF
```

Деталь: при первом запуске собираем `imt_id` для каждого нашего товара
(пройдёмся по `card.wb.ru` или возьмём из ответа `sellers/v4`, проверим).

`🏆 лучшая цена`. Если у нас выше всех — `↑ выше среднего` мягкий warning.

Сколько товаров уже имеют записи `competitor_offers`, сколько нет.

## Открытые вопросы

1. **Работает ли `search.wb.ru` через прод-VPN?** Локально у нас не работает
   (PoW), но это надо проверить на Asha — там может быть другой VPN/прокси.
2. **Какой интервал обновления?** 6 часов? 12? Цены у конкурентов меняются
   реже наших, но мы хотим оперативность.
3. **Когда отрывать конкурентов?** Если конкурент перестал продавать — старая
   запись «висит». Решение: помечать как `stale` если `last_seen_ts > 7 дней назад`.

## Риски

- WB-WAF: при обходе многих товаров получим 429. **Митигация:** медленные
  паузы между запросами + retry с backoff (уже есть в `wb_cffi`).
- `search.wb.ru` может не работать → фича жёстко зависит от endpoint'а.
  **Mitigation:** fallback на ручное добавление пар через админку.
- Дедуп по имени без `sku/imt_id` будет шумный → видеть «MacBook Air 13» в
  списке конкурентов «MacBook Pro 14» очевидно бесит. **Mitigation:** не
  показывать конкурентов с уверенностью якоря ниже 80%.

## Этапы

1. **Phase 1 (3ч):** схема + ручное добавление через админку + API + UI-блок
   в модалке. Без авто-сканирования. Сразу получаем ценность для топ-товаров.
2. **Phase 2 (3ч):** worker-hook + автоматический сбор через `search.wb.ru` или
   `imt_id`. Зависит от того что работает на проде.
3. **Phase 3 (2ч):** «🏆 лучшая цена» бейдж в каталоге + графики «наш vs
   средний» в детальной модалке.
